1. RESUMEN EJECUTIVO
--------------------------------------------------------------------------------
Fit-Daily es una herramienta interna que unifica, en una sola aplicación web, la
gestión ágil (Scrum) del equipo de consultores con el sistema de Helpdesk de
FitBank. Cada tarea del tablero corresponde a un ticket real de un cliente, de
modo que la operación diaria, el seguimiento de avances y la supervisión quedan
conectados de punta a punta.

Valor para el negocio:
  • Una sola pantalla en lugar de saltar entre el Helpdesk y planillas manuales.
  • Visibilidad en tiempo real del trabajo del equipo (tablero Kanban compartido).
  • Trazabilidad: tarea ↔ ticket ↔ cliente ↔ consultor ↔ estado.
  • Panel de supervisión para el Scrum Master (carga, pendientes, bloqueos).
  • Sin costo de infraestructura de servidores: se apoya en servicios gestionados
    (Firebase y Cloudflare) y publicación estática (GitHub Pages).

Estado actual: la aplicación está EN PRODUCCIÓN y en uso por el equipo. Se está
ejecutando una segunda generación construida en Angular 22, que reemplaza
progresivamente a la versión inicial.


2. PROPÓSITO Y ALCANCE
--------------------------------------------------------------------------------
Reemplazar el seguimiento manual de tareas por un tablero Kanban integrado al
Helpdesk. El equipo de consultores atiende a 14 clientes (cooperativas y empresas
financieras que usan el core bancario FitBank). El sistema cubre:

  • Planificación y ejecución de sprints.
  • Tablero Kanban (To Do → En Progreso → En Certificación → Finalizado).
  • Consulta y operación de tickets del Helpdesk (filtrar, asignar, cambiar
    estado, conversar, adjuntar).
  • Panel del Scrum Master, pendientes con recordatorios y asignación semanal.

4. ARQUITECTURA
--------------------------------------------------------------------------------
4.1 Visión general

  La aplicación es una SPA (Single Page Application) que se sirve estáticamente y
  consume DOS fuentes de datos independientes:

     ┌────────────────────────────────────────────────────────────────┐
     │                  NAVEGADOR (App Angular - SPA)                    │
     │  Vistas: Board · Tickets · Mi Panel · Semanal · Pendientes ...   │
     │  Estado en memoria con Signals · Material 3 · responsive          │
     └───────────────┬──────────────────────────────┬───────────────────┘
                     │                                │
         (datos del proyecto)              (tickets / catálogos)
                     │                                │
                     ▼                                ▼
        ┌──────────────────────┐        ┌──────────────────────────────┐
        │  Firebase Realtime DB │        │   Cloudflare Worker (proxy)   │
        │  (sprints, tareas,    │        │            │                  │
        │   avances, notas...)  │        │            ▼                  │
        │  Sync en tiempo real  │        │   API Helpdesk FitBank (REST) │
        └──────────────────────┘        └──────────────────────────────┘

4.2 Frontend
  • Componentes "standalone" con carga perezosa por ruta (cada vista se descarga
    solo cuando se usa → arranque rápido).
  • Estado con Signals: la interfaz se actualiza sola cuando cambian los datos,
    sin código manual de refresco.
  • Capa "core" (servicios) separada de las "features" (vistas):
        core/services   → DataService, AuthService, HelpdeskService, ShellService
        core/interceptors → inyecta el token y maneja expiración (401/403)
        core/guards     → protección de rutas por sesión y por rol
  • Shell responsive (menú lateral) mobile-first con CDK BreakpointObserver.

4.3 Fuentes de datos
  a) Firebase Realtime Database — datos PROPIOS del proyecto (tablero, sprints,
     avances, notas, pendientes). Sincronización en tiempo real entre todas las
     máquinas del equipo (lo que uno cambia, los demás lo ven al instante).
  b) API Helpdesk de FitBank — sistema EXTERNO y fuente de verdad de los tickets,
     usuarios (consultores) y clientes. La app lee y escribe ahí (asignar ticket,
     cambiar estado, responder). Las consultas de tickets se filtran y paginan
     DEL LADO DEL SERVIDOR (por cliente, estado, asignado), no en el navegador,
     para no traer información de más.

4.4 Autenticación y seguridad
  • Inicio de sesión con las credenciales del propio Helpdesk (no hay un usuario
    paralelo). La sesión se guarda localmente.
  • Token de acceso de vida corta + "refresh token": cuando el token expira, la
    app lo renueva automáticamente y reintenta la operación, sin sacar al usuario.
  • Permisos por rol (Consultor / Supervisor / Scrum Master / Helpdesk-admin):
    definen quién puede mover tareas ajenas, ver el panel de supervisión o borrar
    el tablero.
  • El proxy (Cloudflare Worker) evita exponer el API directamente al navegador y
    resuelve las restricciones de CORS.

4.5 Despliegue
  • Publicación estática en GitHub Pages. La app Angular vive en la subcarpeta
    /angular/ y CONVIVE con la versión anterior sin afectarla.
  • Enrutado por hash (#) para que el refresco de página funcione en GitHub Pages.
  • El Worker de Cloudflare se despliega de forma independiente.
  • Flujo: compilar → publicar artefacto → GitHub Pages distribuye por CDN.
  • Auto-actualización (PWA): la app incluye un Service Worker que detecta cada
    versión nueva publicada y la activa sola en el navegador del usuario (recarga
    automática), sin que nadie tenga que borrar la caché y conservando la sesión.

--------------------------------------------------------------------------------
7.1 Roles y permisos
    • Consultor   – opera SUS tareas (mover, avanzar, certificar).
    • Supervisor  – puede gestionar tareas de cualquiera.
    • Scrum Master– acceso al panel de supervisión ("Mi Panel").
    • Helpdesk-admin (cuenta MSC001) – permisos máximos (p. ej. borrar tablero).

7.2 Vistas (agrupadas en menú "Scrum" y "HelpDesk")
    BOARD (Kanban)
      - 4 columnas: To Do · En Progreso · En Certificación · Finalizado.
      - Arrastrar y soltar con permisos y límite de trabajo en progreso (WIP).
      - Tarjetas coloreadas por cliente; badge de prioridad y de estado del ticket.
      - Filtros multi-selección por prioridad, consultor y cliente; búsqueda por
        n° de ticket.
      - Regla de negocio: una tarea que salió de "To Do" no puede regresar; las
        finalizadas desaparecen automáticamente a los 2 días.
      - Sincroniza el estado real del ticket del Helpdesk y ubica la tarea en la
        columna correspondiente; reconcilia la asignación con el API.

    TICKETS (Helpdesk)
      - Grilla de tarjetas de tickets con pestañas: Pendientes · Asignados a mí ·
        Todos los clientes · Estadísticas.
      - Filtros server-side: Cliente (multi-selección con chips removibles),
        Estatus y "Asignados a mí"; ordenamiento por fecha/prioridad.
      - Paginación del lado del servidor con total real ("Página X de Y").
      - Acciones: crear tarea desde el ticket, ver/responder la conversación,
        asignar/reasignar consultor, cambiar estado, notas y marcas internas.

    MI PANEL (Scrum Master)  – supervisión diaria: acciones pendientes, tickets
                               sin movimiento, clientes a notificar.
    SEMANAL                  – asignación semanal de consultor (calendario).
    PENDIENTES               – tickets marcados como pendientes, con recordatorios
                               por fecha/hora (alerta y opción de posponer/pausar).
    BURNDOWN   – métricas del sprint (% completado, en curso, pendientes, días
                 restantes), gráfico de avance y velocidad del equipo por sprint.
    PROGRESO   – registro de avances por historia, con filtros y alta de registro.
    CONSULTAS  – consultas/bloqueos del equipo (abiertas/resueltas), alta y resolución.

7.3 Flujos clave
    • De ticket a tarea: desde un ticket se crea la tarea con datos autocompletados
      (cliente, asunto, asignado) y queda enlazada al ticket.
    • Operación del board: al mover una tarea se escribe el estado correspondiente
      en el ticket del Helpdesk (y viceversa al abrir el board).
    • Supervisión: el Scrum Master ve carga, bloqueos y pendientes sin pedir
      reportes manuales.

A.1 SPA QUE CONSUME DOS FUENTES INDEPENDIENTES
--------------------------------------------------------------------------------
Fit-Daily es una SPA: el navegador descarga la aplicación una sola vez y, a partir
de ahí, la navegación entre pantallas ocurre sin recargar la página (router de
Angular con carga perezosa por vista). El estado vive en memoria con Signals y la
interfaz se deriva de ese estado.

La app se apoya en DOS fuentes de datos desacopladas, cada una con su propio
servicio y su propio mecanismo de acceso:

  a) Firebase Realtime Database — datos PROPIOS del proyecto (sprints, tareas,
     avances, consultas, notas, pendientes). Gestionado por `DataService`.
  b) API Helpdesk de FitBank (vía proxy) — sistema EXTERNO y fuente de verdad de
     los tickets, usuarios (consultores) y clientes. Gestionado por
     `HelpdeskService`.

Beneficio de mantenerlas independientes: el tablero (Firebase) sigue operativo
aunque el Helpdesk no responda, y la vista de tickets funciona aunque el board no
tenga datos. La degradación es aislada, no se cae todo junto. El punto de unión es
lógico (cada tarea referencia el N° de ticket), no técnico.

         Vistas (Signals)
        ┌──────┴───────┐
   DataService     HelpdeskService
        │                │
   Firebase RTDB    Proxy → API Helpdesk

A.2 FIREBASE: SPRINTS, TAREAS, AVANCES (TIEMPO REAL)
--------------------------------------------------------------------------------
Acceso vía REST de Firebase sobre la base `…/fit-daily/<nodo>.json`.

  • Lectura inicial: `ensureInit()` lee los nodos; si están vacíos los SIEMBRA
    desde los JSON locales (semillas) y si ya tienen datos los carga — nunca pisa
    información existente.
  • Escritura: por nodo (PUT/PATCH/DELETE), de forma "fire-and-forget" con captura
    de error (no bloquea la interfaz).
  • TIEMPO REAL (lo más relevante): al iniciar, `startStreaming()` abre un
    EventSource (SSE — Server-Sent Events) sobre el nodo `stories` (las tareas del
    tablero). Cuando Firebase emite un evento `put`/`patch`, la app dispara una
    re-sincronización con "debounce" de 250 ms (agrupa ráfagas de cambios). Como
    respaldo hay un temporizador de seguridad cada 60 s y, si el navegador no
    soporta EventSource, cae a "polling" periódico.
  • Conciliación sin pisarse: al llegar cambios remotos se hace un merge por id de
    tarea, respetando una "ventana de gracia" para las escrituras locales recientes
    (evita que un cambio que el propio usuario acaba de hacer sea sobrescrito por un
    eco del servidor → sin parpadeos ni condiciones de carrera). También elimina en
    local las tareas que ya no existen en remoto.
  • Resultado: lo que un consultor cambia en el board, los demás lo ven casi al
    instante, sin recargar.
  • Respaldo offline: si Firebase no está disponible, `DataService` persiste en
    localStorage como contingencia.

  Nota: el streaming en vivo aplica al nodo `stories`; sprints, avances y consultas
  se cargan al inicio y se refrescan al escribir (no requieren SSE permanente).

A.3 INTERCEPTORES HTTP: TOKEN + RENOVACIÓN AUTOMÁTICA (REFRESH)
--------------------------------------------------------------------------------
Un interceptor HTTP (`helpdeskAuthInterceptor`) procesa TODAS las llamadas al API
del Helpdesk de forma centralizada, sin que cada pantalla tenga que ocuparse de la
autenticación:

  • Inyecta en cada petición la cabecera `Authorization: Bearer <access_token>`
    tomada de la sesión.
  • Renovación automática ante 401 (token vencido): llama a `refreshSession()`, que
    hace `POST /auth/refresh` con el "refresh token"; DEDUPLICA llamadas concurrentes
    (si varias peticiones fallan 401 a la vez, se hace un solo refresh) y, con el
    token nuevo, REINTENTA la petición original. Todo transparente: el usuario no se
    entera ni es expulsado.
  • Si el refresh también falla (refresh token vencido) o ante un 403 (sin permiso),
    cierra la sesión y redirige a /login.
  • Marca de peticiones "tolerables" (HD_SAFE): catálogos y escrituras best-effort
    que, si fallan, no fuerzan el cierre de sesión.
  • Diseño a prueba de bucles: el reintento no vuelve a entrar al manejo de error y
    las rutas de autenticación quedan excluidas.

  Beneficio: sesiones de trabajo largas sin cortes, con tokens de acceso de vida
  corta (más seguros) renovados de forma silenciosa.

A.4 GUARDS DE RUTAS POR SESIÓN Y POR ROL
--------------------------------------------------------------------------------
El acceso a cada pantalla se controla con "guards" (porteros) de ruta:

  • authGuard (sesión): protege TODAS las vistas bajo el shell. Si no hay sesión
    activa, redirige a /login.
  • solGuard (rol): la vista "Mi Panel" (supervisión) solo la abren el Scrum Master
    o el administrador del Helpdesk (MSC001); el resto es redirigido al tablero.
  • msc001Guard (rol): reservado para acciones exclusivas del administrador
    (p. ej. borrar el tablero completo).

Los permisos (esMSC001, esSupervisor, puedeVerMiPanel, puedeBorrarBoard,
puedeGestionarTodo) son valores derivados del ROL que el API del Helpdesk asigna al
usuario (role_description) y que viaja en la sesión.

Defensa en capas: además del guard que bloquea la navegación, dentro de cada vista
se valida el permiso por acción concreta (por ejemplo, en el tablero solo el
asignado, un Supervisor o el administrador pueden mover una tarea ajena o marcar sus
checks; a quien no tiene permiso se le muestra un aviso y el cambio no se aplica).
